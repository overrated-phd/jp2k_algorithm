JPEG 2000
End-to-End Explained

A mathematically rigorous, interactive exploration of the wavelet-based image compression standard — from lifting-scheme DWT through EBCOT entropy coding, including bit-plane & pass parallel hardware architectures (Miyama et al., Kanazawa University), with working simulations and machine learning connections.

§1 — Overview

What is JPEG 2000?

A wavelet-based image compression standard superseding JPEG, offering superior rate-distortion, lossless capability, and progressive transmission in a single unified codestream.

JPEG 2000, standardized in 2000 by the Joint Photographic Experts Group, replaces JPEG's 8×8 block Discrete Cosine Transform (DCT) with a full-frame Discrete Wavelet Transform (DWT). This eliminates blocking artifacts at low bitrates and enables progressive decoding, Region of Interest (ROI) coding, and unified lossless/lossy compression.

The standard spans multiple parts: Part 1 (core codec), Part 2 (extensions), Part 3 (Motion JPEG 2000), Part 9 (JPIP interactive protocol), and Part 15 (HTJ2K high-throughput). Primary domains include digital cinema (DCI mandates JP2K), medical imaging (DICOM lossless archival), satellite/geospatial (Sentinel-2, GOES), and real-time video transmission such as wireless PC displays.

Key Innovation: Embedded Coding

JP2K produces a single embedded bitstream truncatable at any point to yield the optimal image at that rate. This is enabled by EBCOT's rate-distortion optimized layering. However, EBCOT is also the computational bottleneck — motivating hardware architectures like the bit-plane and pass parallel approach by Miyama et al.

Primary hardware reference: M. Miyama, Y. Inoie, T. Kasuga, R. Inada, R. Kawai, M. Nakao, Y. Matsuda, "A 158 MS/s JPEG 2000 Codec with a Bit-Plane and Pass Parallel Embedded Block Coder," Kanazawa University / Eizo Nanao Corporation.

§3 — Mathematical Foundations

Stage-by-Stage Analysis

3.1 Tiling

The image W × H is partitioned into non-overlapping tiles of size T_w × T_h. The Miyama et al. codec uses 128 × 128 tiles, balancing memory requirements against seam artifacts. Each tile is compressed independently, bounding memory to O(T_w · T_h).

3.2 DC Level Shift

DC Level Shiftx'(i, j) = x(i, j) − 2B−1

For B = 8, this maps [0, 255] → [−128, 127], centering the signal for improved DWT efficiency.

3.3 Color Transform

Irreversible Color Transform (ICT) — Lossy Mode
Forward ICT (RGB → YCbCr)Y = 0.299·R + 0.587·G + 0.114·B
Cb = −0.16875·R − 0.33126·G + 0.5·B
Cr = 0.5·R − 0.41869·G − 0.08131·B
Reversible Color Transform (RCT) — Lossless Mode
Forward RCT (integer-only)Y = ⌊(R + 2G + B) / 4⌋
Cb = B − G
Cr = R − G
Inverse RCTG = Y − ⌊(Cb + Cr) / 4⌋, R = Cr + G, B = Cb + G

Integer division guarantees bit-exact invertibility for lossless compression.

In the Miyama et al. codec, the CCNV (Color Converter) module implements RGB↔YCbCr bidirectionally with reconfigurable logic to share gates between encoder and decoder.

3.4 Discrete Wavelet Transform

The DWT decomposes the image via separable 1-D filtering into four subbands per level:

LL

Approximation

Low-pass both directions. Recursively decomposed at subsequent levels.

LH

Horizontal Detail

Low-pass horizontal, high-pass vertical. Captures horizontal edges.

HL

Vertical Detail

High-pass horizontal, low-pass vertical. Captures vertical edges.

HH

Diagonal Detail

High-pass both directions. Captures diagonal textures and noise.

3.5 The Lifting Scheme

Le Gall 5/3 — Lossless Integer Lifting
Split → Predict → Updatesj(0) = x(2j), dj(0) = x(2j + 1)
dj = dj(0) − ⌊(sj(0) + sj+1(0)) / 2⌋
sj = sj(0) + ⌊(dj−1 + dj + 2) / 4⌋
CDF 9/7 — Lossy Real-Valued Lifting (4 steps)

d(2) = d(1) + γ·(sj(1) + sj+1(1)) γ = 0.882911075

Normalizations = K · s(2), d = (1/K) · d(2), K = 1.230174105

3.7 Dead-Zone Scalar Quantization

Forward quantizationqb(i,j) = sign(ab(i,j)) · ⌊ |ab(i,j)| / Δb ⌋

Dead zone width is 2Δ_b, aggressively zeroing small coefficients.

Step size from header parametersΔb = 2Rb − εb · (1 + μb / 211)

εb ∈ {0,…,31} (5 bits), μb ∈ {0,…,2047} (11 bits)
Dequantizationâb(i,j) = Δb · (qb(i,j) + sign(q)·δ) for q ≠ 0, δ = 0.5 typically

In the Miyama et al. codec, the Q module performs both quantization and inverse quantization using reconfigurable logic, sharing hardware between encoding and decoding directions.

3.8 EBCOT — The Computational Bottleneck

EBCOT is the most complex stage of JPEG 2000. Each subband is partitioned into code-blocks (typically 32×32 in the Miyama codec, or 64×64 in standard implementations). Within each code-block, coefficients are coded bitplane by bitplane from MSB to LSB.

Significance State

A coefficient becomes significant on the bitplane where its first '1' appears when scanning from the MSB. This state propagates information to lower bitplanes and to neighboring coefficients — creating the dependency chain that makes parallelization challenging.

Three Coding Passes per Bitplane

Significance Propagation (SP): Encodes not-yet-significant samples that have ≥1 significant 8-connected neighbor. The pass determination depends on significance states of all 8 surrounding coefficients.

Magnitude Refinement (MR): Refines previously significant samples. Crucially, MR does not change the significance state — this property is exploited for parallelism.

Cleanup (CU): Encodes all remaining samples via run-length + MQ coding.

3.9 Context Formation

Significance context labelκsig = f(σh, σv, σd)

σh = # significant horizontal neighbors ∈ {0,1,2}
σv = # significant vertical neighbors ∈ {0,1,2}
σd = # significant diagonal neighbors ∈ {0,…,4}

Neighbor layout around coefficient at position (i,j):

This 8-neighbor dependency is what makes the regular mode harder to parallelize than the vertically causal mode. In vertically causal mode, coefficients g, h, i (across stripe boundaries) are excluded from context formation, simplifying dependencies at a cost to image quality.

§5 — Parallel EBCOT Architecture

Bit-Plane & Pass Parallel Embedded Block Coder

Based on M. Miyama et al. (Kanazawa University / Eizo Nanao Corp.) — a hardware architecture achieving 158 MS/s by parallelizing EBCOT across all bitplanes and all coding passes simultaneously.

5.1 The EBCOT Bottleneck Problem

In conventional EBCOT, bitplanes are coded sequentially from the MSB to the LSB. Within each bitplane, the three passes (SP → MR → CU) also execute sequentially. For a code-block with M bitplanes, each with 3 passes, the total latency is 3M × T_pass — making EBCOT the throughput bottleneck of the entire JPEG 2000 pipeline.

Conventional EBCOT latencyTconventional = Σm=0M−1 [ TSP(m) + TMR(m) + TCU(m) ]

All bitplanes and passes are strictly sequential.
For 10-bit coefficients with 32×32 code-blocks: M=10, ~30 sequential passes.

5.2 Bit-Plane Parallel Method

The key insight: bitplane m only depends on significance states computed on bitplane m+1. If the upper bitplane has a sufficient head start, the lower bitplane can begin coding while the upper is still in progress.

Bit-Plane Overlap with 1-Stripe, 1-Column Gap

All M bitplanes in a code-block are coded in parallel. An upper bitplane and a lower bitplane overlap in time with a gap of one stripe (4 rows) and one column. This gap is necessary because, in the regular mode, the significance state of bit i on the upper bitplane is needed for bit-modeling of bit e on the lower bitplane — where i is the diagonal neighbor one stripe and one column ahead.

Neighbor dependency (regular mode)For coefficient at position (r, c) on bitplane m:
Requires significance of (r+1, c+1) on bitplane m+1

Layout: a b c ← stripe n-1
         d e f ← stripe n (current)
         g h i ← stripe n+1

Bit 'e' on bitplane#m needs state of bit 'i' from bitplane#(m+1)
→ 1-stripe + 1-column gap enables regular mode support

In vertically causal mode, neighbors g, h, i (across stripe boundaries) are excluded, so only a 1-column gap would suffice. However, regular mode provides better image quality, and the Miyama architecture supports both modes with the unified 1-stripe + 1-column gap.

5.3 Pass Parallel Method

Within each bitplane, the three coding passes (SP, MR, CU) are also overlapped in time with the same 1-stripe, 1-column gap:

Pass Overlap Mechanism

For each stripe of 8 scanning bits: the pass determination step classifies bits into SP, MR, or CU. The SP pass scans the lower-right 4 bits while the MR+CU passes scan the upper-left 4 bits from the previous determination. The MR and CU passes can scan the same 4 bits simultaneously because the MR pass does not change the significance state — a crucial property that enables this parallelism.

Pass determination for 8 bits in a stripe columnStep k: SP scans bits {4,5,6,7} from stripe n
         MR+CU scan bits {0,1,2,3} from stripe n

MR ∩ CU parallelism is safe because:
MR only refines magnitude of already-significant coefficients
→ significance state σ(i,j) is unchanged by MR
→ CU context formation is not affected

5.5 EBCOT Hardware Architecture

The EBCOT consists of three main components:

CBB

Code-Block Buffer

Stores quantized coefficients by code-block. Each bit-plane in a code-block is accessible in parallel — enabling all BPCs to read simultaneously.

EBC

Embedded Block Coder

Contains M Bit-Plane Coders (BPCs), one per bitplane. BPCs connect via FIFOs — upper BPC sends significance states and sign bits to lower BPC for self-synchronization.

CSB

Code-Stream Buffer

Stores coded bitstream for each pass on each bit-plane. Divided into areas corresponding to each bitplane for independent access.

5.6 Bit-Plane Coder (BPC) Internal Architecture

Each BPC processes one bitplane and contains the following pipeline stages:

BPC Pipeline Stages

SSB (Significance State Buffer): Stores significance states, sign bits, and quantized coefficients for three stripes only (not all stripes), reducing memory. Significance states use 2-bit encoding instead of the conventional 3-bit — states {0: not significant, 1: significant but not MR-scanned on same bitplane, 2: significant but not first-MR-modeled, 3: significant and first-MR-modeled}.

PP (Pre-Processor): Determines the pass (SP/MR/CU) for each bit and sends neighbor states to FIFO.

CXD (Context Generator): Generates (context, decision) pairs from neighbor states. At decoding, uses data from both PP and UPD module.

MQC / MQD: MQ arithmetic encoder and decoder respectively. The MQD incorporates speculative decoding logic.

UPD (Update): Updates significance states using decoded decisions. Handles both "past" states (from decoded bits) and "future" states (from PP speculation).

SSW (State Writer): Writes back to SSB. At decoding, decoded states from MQD overwrite speculative states from PP.

FIFO-based self-synchronizationBPC[m+1] →FIFO BPC[m] →FIFO BPC[m−1] → ··· → BPC[0]

Data transferred per coefficient: {significance_state, sign_bit}
FIFO naturally handles rate mismatch between upper and lower BPC.
Upper coding always finishes before lower due to significance dependency.
No external synchronization controller needed.

5.7 Speculative Decoding

The decoder faces a critical problem: at decoding, the neighbor significance states needed for context formation are only known after decoding the previous bit — creating a serial dependency. The Miyama architecture breaks this with two speculative techniques:

Speculation Method 1: SP Pass Pre-computation

For bits possibly modeled by the SP pass (bits neighboring a significant coefficient, but whose own state and all neighbors are not yet significant), the PP speculatively sends neighbor states to FIFO before the previous bit finishes decoding. If the previous bit becomes significant after decoding, the MQD proceeds with actual decoding. If not, MQD informs SSW that the bit was not SP-modeled (deferred to CU pass).

Speculation Method 2: Successive-Bit Context Prediction

When two successive bits of the same pass are decoded, the CXD generates the lower bit's context assuming the upper bit is not significant. If the upper bit turns out to be significant, the context is modified inside the MQ decoder — not regenerated from scratch. This avoids re-reading neighbor states and keeps the pipeline flowing.

Speculative context pipelineStage S0: Pass Determination (bit n)
Stage S1: CX Generation (bit n) — speculative, assumes bit n−1 non-significant
Stage S2: MQ Decode (bit n) — may modify context if bit n−1 was significant
Stage S3: Update (bit n)

Without speculation: pipeline stalls at S1 waiting for S2 of bit n−1
With speculation: 1 sample per cycle throughput achieved

The key insight is that modifying a context within the MQ decoder is simpler than regenerating the true context from 8 neighbor significance states. Re-calculation and MQ decoding can fit in the same pipeline stage, so the decode pipeline never stalls.

5.8 2-Bit Significance State Encoding

Conventional implementations use 3 bits for significance state. The Miyama architecture reduces this to 2 bits, cutting SSB memory by 33%:

Optimized 2-bit significance statesState 0: Not significant
State 1: Significant, not yet scanned by MR on the bitplane where SP made it significant
State 2: Significant, not yet modeled by the first MR pass
State 3: Significant, has been modeled by the first MR pass

§7 — Machine Learning Era

JPEG 2000 in Modern AI Pipelines

JP2K intersects ML as both competitor (learned compression) and complement (efficient features, domain-specific workflows).

1 Learned Image Compression vs. JP2K

Neural codecs (Ballé 2018, Cheng 2020, ELIC 2022) replace DWT + EBCOT with end-to-end learned components:

Neural Codec Frameworky = ga(x; θ) — analysis transform (CNN/ViT)
ŷ = Q(y) — quantization
x̂ = gs(ŷ; φ) — synthesis transform

Rate-Distortion LossL = 𝔼[−log₂ pŷ(ŷ)] + λ·‖x − x̂‖²

Neural codecs save 20–40% bitrate over JP2K, but require GPU, lack progressive/ROI capability, and cannot guarantee lossless compression. High-throughput hardware JP2K (158 MS/s) remains competitive for latency-critical applications like the wireless PC display targeted by the Miyama codec.

2 DWT as Neural Network Feature Extraction

DWT ≡ Fixed Conv + Stride-2LL = (hlo ⊛ x) ↓2 ≡ Conv2d(C, C, k=5, stride=2, weights=hlo)

DWT preserves spatial + frequency info (invertible)
MaxPool discards position + high-freq (irreversible)

WaveCNet (Li 2020) replaces pooling with DWT for noise-robust classification. Wavelet Scattering (Mallat 2012) provides provably Lipschitz features.

JPIP enables this workflow natively. The Miyama codec's regular mode support ensures no quality degradation from vertically causal artifacts in the transmitted subbands.

6 Hardware-Accelerated JP2K in ML Pipelines

The Miyama codec's 158 MS/s throughput at 1.5W represents a compelling alternative to GPU-based neural codecs in edge ML scenarios. For embedded vision systems (autonomous vehicles, drones, medical devices), JP2K hardware provides deterministic latency, low power, and standardized interoperability that neural codecs cannot match. The regular mode support ensures diagnostic-quality encoding for medical AI inference, while the JPIP-compatible progressive bitstream enables bandwidth-adaptive transmission for remote ML inference servers.

Code: DWT Pooling Layer for PyTorch

wavelet_pooling.py — drop-in replacement for MaxPool2d
import torch, torch.nn as nn, pywt class DWTPooling(nn.Module): """Replace MaxPool2d with DWT (bior4.4 ≈ CDF 9/7). Output: 4× channels, half spatial resolution.""" def __init__(self, wavelet='bior4.4'): super().__init__() w = pywt.Wavelet(wavelet) self.register_buffer('lo', torch.tensor(w.dec_lo, dtype=torch.float32)) self.register_buffer('hi', torch.tensor(w.dec_hi, dtype=torch.float32)) def _filt(self, x, fr, fc): B,C,H,W = x.shape; K=fr.shape[0]; p=K//2 xp = nn.functional.pad(x, [p,p], mode='reflect') r = nn.functional.conv2d(xp, fr[None,None,:,None].expand(C,1,-1,-1), groups=C)[..., ::2] rp = nn.functional.pad(r, [0,0,p,p], mode='reflect') return nn.functional.conv2d(rp, fc[None,None,None,:].expand(C,1,-1,-1), groups=C)[:,:,::2,:] def forward(self, x): return torch.cat([ self._filt(x, self.lo, self.lo), # LL self._filt(x, self.lo, self.hi), # LH self._filt(x, self.hi, self.lo), # HL self._filt(x, self.hi, self.hi), # HH ], dim=1)
